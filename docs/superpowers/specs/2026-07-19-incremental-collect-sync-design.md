# 增量采集同步(水位线 + 批量写入)设计

日期:2026-07-19
状态:已确认

## 背景与问题

数据库迁移到远程(Supabase Postgres)后,每次采集的写库方式暴露出严重 IO 问题:

- `time_usage`:每个数据源先 `DELETE` 该 (device, source) 的全部行,再把 90 天窗口内的事件(当前约 11,500 行)逐行 `INSERT`。
- `daily_usage`(约 379 行)与 `session_usage`(约 618 行):全量逐行 upsert。
- 每条语句一次网络往返,单次采集超过 1.2 万次往返,对新加坡节点实测约 30 分钟,并在 Postgres 侧产生大量 WAL 与死元组。

根因:`eventKey` 拼入了数组下标 `index`(`src/collect.mjs`),事件顺序变化导致 key 不稳定,无法幂等 upsert,只能删表重插。

## 相关既有约定:历史价格锁定

提交 `f98dd4f`(2026-06-10)已实现 `daily_usage` 的价格锁定:昨天及更早的行 `cost_usd` 不接受重算,只有当天的行按当前价格更新(`pricing_locked_at` 字段)。

**缺口**:`time_usage` 因删表重插,老事件成本每次都按当前价格重算,与日汇总视图不一致。本设计的增量写入(老事件永不重写)自动使 `time_usage` 获得相同的锁定语义,无需新增锁字段。

## 设计

### 1. 稳定化 eventKey

- 去掉 key 中的数组下标 `index`。
- 新 key = `client:eventTime:sessionId:model:totalTokens`,批内相同元组按出现顺序追加序号(`#0`、`#1`、…)。
- 同一份源日志重复采集生成完全相同的 key 集合,upsert 幂等。
- key 规则变更意味着与旧数据不兼容,首次上线依赖 `--full` 重建(见第 4 节)。

### 2. 水位线增量写入

- 采集开始时按 (device, source) 查询远端 `MAX(event_time)` 作为水位线。
- `time_usage`:只上传 `eventTime > 水位线 − 48h` 的事件;重叠窗口容忍日志晚写,幂等 upsert 保证重叠部分不重复。**不再执行全表 DELETE。**
- `daily_usage`:只 upsert `usageDate >= (水位线 − 48h) 的日期` 的行;更早日期的行 token 与成本均不再变化(成本本就被锁定)。
- `session_usage`:累计值每次都变,保持全量,但走批量写。
- 远端该 (device, source) 无数据时(水位线为空)自动走全量路径。

### 3. 批量写入

- 三张表的逐行 upsert 改为多行 `INSERT ... VALUES (...),(...) ON CONFLICT DO UPDATE`,每批 500 行。
- SQLite 与 Postgres 语法一致,MySQL 用 `ON DUPLICATE KEY UPDATE` 多行等价形式;本地与远端同一套代码路径。
- 网络往返从 1.2 万次降至 30 次以内。

### 4. `--full` 全量重建

- `node src/collect.mjs --full`:保留"删除 + 重传"语义(同样批量化),用于 key 规则变更、采集器 bug 修复等结构性重建。
- 明确语义:全量重建会按**当前**价格重算历史事件成本,是有意为之的重置,不是常规操作;价格表日常更新不需要 `--full`。

### 5. 语义变化:远端保留超窗历史

去掉全表删除后,本地日志滚动清理(超过 90 天)的事件在远端仍保留。视为功能升级(历史更完整),不做清理任务。

## 测试

- key 稳定性:同一输入两次生成相同 key 集合;重复元组序号正确。
- 水位线过滤:边界(恰好等于水位线、窗口内、窗口外)行为正确;空远端走全量。
- 批量 SQL 构造:分批边界(0 行、1 行、500 行、501 行);三种驱动的语法分支。
- 集成:本地 SQLite 全量采集 → 增量采集,前后行数与聚合值一致;增量不改动老事件 cost_usd。
- 现有 `node --test` 全部通过。

## 非目标

- 不做逐条脏标记/内容哈希变更检测(全局性变更由 `--full` 覆盖,零散变更不存在)。
- 不做远端超窗数据清理。
- 不做定时采集(用户当前手动触发)。
